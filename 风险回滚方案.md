# Aurora 重构风险与回滚方案

> 本文档补充 重构方案.md 第七节的风险列表，为每个高风险步骤提供具体的回滚操作方案。

---

## 一、总体原则

1. **原子化提交**：每次改动只涉及一个模块/一个方面，确保可以部分回滚。
2. **基线标记**：Git 中每个 Phase 完成时打 Tag（如 `phase-1-done`），便于快速回退。
3. **保留原版**：`aurora/` 目录始终保持原始代码不变，作为最后回退点。
4. **先测试，再提交**：每次提交前保证后端能编译、核心接口可访问。

---

## 二、风险矩阵与回滚预案

### 2.1 Risk-1：Spring Boot 2.3.7 → 3.4+ 启动失败

**概率**：高
**影响**：后端完全不可用

**回滚步骤**：
```
1. git checkout -- pom.xml   # 恢复原始 POM
2. git checkout -- src/main/resources/   # 恢复配置文件
3. 手动撤销 javax → jakarta 的代码修改：
   - 在 IDEA 中打开 Local History 逐文件恢复
   - 或 git checkout -- 指定文件
4. mvn clean compile 验证恢复后可编译
5. 检查 Spring Boot 版本回到 2.3.7.RELEASE
```

**预防措施**：
- 先在分支上实验 Spring Boot 3.x 升级
- 使用 Spring Boot Migrator 工具扫描迁移点
- 参考官方迁移指南 https://github.com/spring-projects/spring-boot/wiki/Spring-Boot-3.0-Migration-Guide

---

### 2.2 Risk-2：JDK 25 编译/运行不兼容

**概率**：中
**影响**：后端编译失败或运行时异常

**诊断检查清单**：
```
1. java -version → 确认是 JDK 25
2. mvn clean compile → 检查编译错误
3. 常见原因：
   - Lombok 版本不足 → 确保 1.18.34+
   - JavaFX/模块化依赖冲突 → 检查 --add-opens
   - 第三方库使用废弃 API → 查看编译警告
```

**回滚步骤**：
```
1. mv pom.xml pom.xml.bak && git checkout pom.xml  # 恢复 POM
2. 将 JAVA_HOME 切换回 JDK 1.8（如果保留）
3. 或用 Docker 运行 JDK 25 容器，与本机环境隔离
```

**预防措施**：
- 先在 Docker 中验证 JDK 25 兼容性
- 使用 GitHub Actions 的 matrix 策略同时测试 JDK 21 和 JDK 25

---

### 2.3 Risk-3：Elasticsearch 7.9.2 → 8.15.x 升级后搜索不可用

**概率**：高
**影响**：前台搜索功能不可用，文章索引丢失

**回滚步骤**：
```
1. 停止 ES 8.15.x 服务
2. 恢复 ES 7.9.2 服务：
   - 确保 ES 7.9.2 的 data 目录未被删除
   - 或从快照恢复
3. 回滚 pom.xml 中 Spring Data Elasticsearch 版本
4. 回滚 application.yml 中的 ES 配置（url、端口）
5. 重启后端验证搜索接口可用
```

**预防措施**：
- ES 升级前对原有索引做快照备份
- 保留 ES 7.9.2 的安装包和数据目录
- 升级后需要重建索引（通过 Maxim 重新同步或手动触发）

---

### 2.4 Risk-4：Spring Security 6.x 改造后权限异常

**概率**：高
**影响**：登录失败或权限判断错误

**诊断检查清单**：
```
1. POST /users/login → 检查能否正常登录
2. 检查返回的 JWT 是否能通过 parseToken 解析
3. 携带 Token 访问受保护接口 → 检查状态码
4. 检查控制台是否有 Security 相关异常
```

**回滚步骤**：
```
1. git checkout -- src/main/java/com/aurora/config/WebSecurityConfig*
2. git checkout -- src/main/java/com/aurora/filter/JwtAuthenticationTokenFilter*
3. git checkout -- src/main/java/com/aurora/service/impl/TokenServiceImpl*
4. 重启并验证登录可用
```

**预防措施**：
- Security 改造分步进行：先升级依赖不改造代码，再逐模块迁移
- 保留旧 Security 配置的备份
- 使用 @Profile 在切换期间保留两套配置

---

### 2.5 Risk-5：javax → jakarta 替换遗漏

**概率**：中
**影响**：编译报错，运行时 NoClassDefFoundError

**诊断方法**：
```
mvn clean compile 2>&1 | findstr "javax".md
# 或使用 IDE 全局搜索 "import javax."
```

**修复而非回滚**：
- 逐模块搜索 `import javax.*`，替换为 `import jakarta.*`
- 重点检查：HttpServletRequest/Response、@PostConstruct/@PreDestroy、@Valid 等

---

### 2.6 Risk-6：fastjson 替换为 Jackson 后序列化行为不一致

**概率**：中
**影响**：特定 API 返回格式异常（时间格式、空值处理、泛型反序列化）

**回滚步骤**：
```
1. 在 pom.xml 中将 fastjson 重新加入依赖
2. 恢复使用 fastjson 的工具类（JSONUtil 等）
3. 逐步替换单个接口，而非全局替换
```

**预防措施**：
- 先在测试环境验证核心接口的 JSON 输出
- 统一 Jackson 配置（日期格式、时区、空值序列化）
- 为关键接口补充集成测试

---

### 2.7 Risk-7：前端 Node.js 24.x 构建失败

**概率**：高
**影响**：前端无法启动/构建

**诊断检查清单**：
```
1. node -v → 确认是 v24.x
2. npm install --legacy-peer-deps
3. $env:NODE_OPTIONS="--openssl-legacy-provider"; npm run serve
4. 查看控制台具体错误信息
```

**解决步骤（非回滚）**：
```
1. 删除 node_modules 和 package-lock.json，重新安装
2. 升级 @vue/cli-service 到 5.0.8
3. 将 node-sass 替换为 sass（dart-sass）
4. 添加 .npmrc 文件：legacy-peer-deps=true
5. 如仍失败，考虑用 Docker 运行 Node 18 版本作为临时方案
```

## 三、快速回滚命令集

### 3.1 回滚特定文件

```powershell
# 回滚单个文件到初始基线
git checkout HEAD -- 文件路径

# 回滚整个模块
git checkout HEAD -- aurora-springboot/pom.xml
git checkout HEAD -- aurora-springboot/src/main/java/com/aurora/config/
git checkout HEAD -- aurora-springboot/src/main/resources/
```

### 3.2 回滚到特定版本

```powershell
# 查看提交历史
git log --oneline

# 回滚到某次提交（保留后续修改为未暂存状态）
git reset --soft <commit-hash>

# 彻底回滚到某次提交（丢失后续修改）
git reset --hard <commit-hash>
```

### 3.3 分支切换策略

```powershell
# 在 Phase 1 开始前创建分支
git checkout -b phase-1-upgrade

# 如果 Phase 1 彻底失败，回退到 main
git checkout main
git branch -D phase-1-upgrade
```

## 四、回滚决策矩阵

| 场景 | 继续修复 | 部分回滚 | 完全回滚 |
|------|---------|---------|---------|
| JDK 版本问题 | 推荐 | 可选 | 不推荐（JDK 25 锁定） |
| Spring Boot 编译失败 | 可选 | 推荐 | 备选 |
| ES 搜索不可用 | 可选 | 推荐 | 备选 |
| Security 权限异常 | 推荐 | 推荐 | 备选 |
| 前端编译失败 | 推荐 | 备选 | 不推荐 |
| 数据库连接失败 | 推荐 | 备选 | 不推荐（保留 MySQL 9.x） |

> 说明：推荐=优先尝试此方案，备选=前两个方案无效时使用，不推荐=尽量避免。

## 五、环境快照与备份建议

| 组件 | 备份内容 | 备份方式 |
|------|---------|---------|
| MySQL | 数据库全量导出 | `mysqldump -u root -p aurora > aurora_backup.sql` |
| ES 索引 | 索引快照 | ES snapshot API 或数据目录复制 |
| Redis | RDB 快照 | `redis-cli SAVE` + 复制 dump.rdb |
| 配置文件 | application-*.yml | Git 提交前手动复制 |

> 建议在 Phase 1 开始前对 MySQL 和 ES 做全量备份。
