## 一、项目概述

### 1.1 项目简介

Aurora 是一个基于前后端分离架构的个人博客系统，支持文章发布与管理、分类标签、评论互动、Elasticsearch 全文搜索、相册管理、说说动态、友链管理、RBAC 权限控制等完整功能。项目前台面向访客提供博客浏览体验，后台面向管理员提供内容管理界面。

### 1.2 项目规模

| 维度 | 数据 |
|------|------|
| 后端 Java 文件 | ~140+ 个（含 Controller、Service、Mapper、Entity、Config 等） |
| 数据库表 | 24 张 |
| REST API 端点 | ~100+ 个 |
| 前台 Vue 组件 | ~30+ 个 |
| 后台管理页面 | ~18 个模块 |
| 核心业务模块 | 文章、评论、用户、分类、标签、相册、说说、友链、权限、定时任务、日志 |

## 二、重构背景与动机

### 2.1 原始项目技术债

原始项目（`aurora/`）建于 2022-2023 年，技术栈停留在：

- **JDK 1.8**：已超过官方维护期限（2022 年 3 月 EOL），无法使用 JDK 9-25 的语言特性（Records、Pattern Matching、Sealed Classes、Virtual Threads 等）
- **Spring Boot 2.3.7.RELEASE**：已结束 OSS 支持（2021 年），存在已知 CVE 漏洞
- **Spring Security 5.x**：依赖 `WebSecurityConfigurerAdapter`（已在 6.x 中删除）
- **javax.\* 命名空间**：Jakarta EE 已将命名空间迁移至 `jakarta.*`，`javax.*` 在 Spring Boot 3.x 中不再兼容
- **Elasticsearch 7.9.2**：bundled JDK 版本过低，无法在 JDK 25 上运行
- **jjwt 0.9.0**：存在已知安全问题，API 已过时
- **fastjson 1.2.76**：历史 CVE 漏洞较多
- **UserAgentUtils 1.20**：自 2015 年起停更

### 2.2 重构驱动力

1. **安全性**：消除 fastjson、jjwt、Spring Boot 2.3.x 等已知漏洞
2. **可维护性**：升级到主流版本，降低新开发者上手成本
3. **性能**：利用 JDK 25 和 Spring Boot 3.x 的性能优化（Virtual Threads、NUMA-aware GC 等）
4. **技术演进**：`javax.*` → `jakarta.*` 是 Java 生态不可逆的趋势
5. **工程化**：引入 Docker Compose、统一配置管理、测试体系

### 2.3 重构原则

1. **保持业务功能不变**：重构期间不新增业务功能
2. **原始项目只读**：`aurora/` 目录作为对比基线，禁止修改
3. **分阶段提交**：每个 Part 独立提交，便于回滚
4. **先跑起来再优化**：先完成核心升级，再进入代码治理
5. **避免过度设计**：不引入不必要的抽象

### 5.2 Spring Security 6.x + JWT 认证

重构的核心变动之一。Spring Security 6.x 废弃了 `WebSecurityConfigurerAdapter` 抽象类，改为基于 `SecurityFilterChain` Bean 的 Lambda DSL 配置。

```java
@Configuration
@EnableWebSecurity
@EnableMethodSecurity
public class WebSecurityConfig {

    @Bean
    public SecurityFilterChain filterChain(HttpSecurity http) throws Exception {
        return http
            .csrf(AbstractHttpConfigurer::disable)
            .cors(Customizer.withDefaults())
            .sessionManagement(session ->
                session.sessionCreationPolicy(SessionCreationPolicy.STATELESS))
            .authorizeRequests(auth -> auth  // 建议升级为 authorizeHttpRequests
                .requestMatchers(URL_WHITELIST).permitAll()
                .anyRequest().authenticated())
            .authenticationProvider(authenticationProvider())
            .addFilterBefore(jwtAuthenticationTokenFilter, UsernamePasswordAuthenticationFilter.class)
            .build();
    }
}
```

**JWT 认证流程：**

```
用户登录
  ↓ UserAuthService 校验凭据
  ↓ TokenService.createToken() 生成 JWT（HS256）
  ↓ Redis hSet(login_user, userId, userInfo)
  ↓ 返回 JWT + UserInfoDTO
  ↓
每次请求
  ↓ JwtAuthenticationTokenFilter
  ↓ 提取 Authorization: Bearer <token>
  ↓ 解析 Claims → 从 Redis 查用户 → 构建 Authentication
  ↓ SecurityContextHolder.getContext().setAuthentication(auth)
  ↓ 放行到后续 Filter / Controller
```

**jjwt 0.12.x API 变更（与 0.9.x 对比）：**

| 操作 | jjwt 0.9.0 | jjwt 0.12.6 |
|------|-----------|------------|
| 生成 Token | `Jwts.builder().setSubject(s).signWith(SignatureAlgorithm.HS256, key)` | `Jwts.builder().subject(s).signWith(key, Jwts.SIG.HS256)` |
| 解析 Token | `Jwts.parser().setSigningKey(key).parseClaimsJws(token).getBody()` | `Jwts.parser().verifyWith(key).build().parseSignedClaims(token).getPayload()` |
| 密钥要求 | 任意长度 | 256-bit（自动通过 SHA-256 摘要满足） |

### 5.3 MyBatis-Plus 3.5.9

**关键变更：**
- 使用 `mybatis-plus-spring-boot3-starter`（对应 Spring Boot 3.x）
- `PaginationInterceptor` → `MybatisPlusInterceptor + PaginationInnerInterceptor`
- `PaginationInnerInterceptor` 从 3.5.9 起拆分到独立模块 `mybatis-plus-jsqlparser`
- `selectCount()` 返回类型从 `Integer` 变为 `Long`，需 `.intValue()` 转换

```java
@Configuration
public class MybatisPlusConfig {
    @Bean
    public MybatisPlusInterceptor mybatisPlusInterceptor() {
        MybatisPlusInterceptor interceptor = new MybatisPlusInterceptor();
        interceptor.addInnerInterceptor(new PaginationInnerInterceptor(DbType.MYSQL));
        return interceptor;
    }
}
```

### 5.4 Elasticsearch 8.15.x 升级

这是整个重构中 API 变化最大的一环。Spring Data Elasticsearch 5.x 底层从 `org.elasticsearch.*`（Transport Client 时代的 API）全面切换到 `co.elastic.clients.*`（Elasticsearch Java API Client 8.x）。

**核心 API 映射：**

| 旧 API (Spring Data ES 4.x / ES 7.x) | 新 API (Spring Data ES 5.x / ES 8.x) |
|------|------|
| `ElasticsearchRestTemplate` | `ElasticsearchOperations` |
| `NativeSearchQueryBuilder` | `NativeQuery.builder()` |
| `org.elasticsearch.index.query.QueryBuilders` | `co.elastic.clients.elasticsearch._types.query_dsl.*` |
| `org.elasticsearch.search.fetch.subphase.highlight.HighlightBuilder` | `HighlightQuery + Highlight + HighlightField + HighlightFieldParameters` |

**ES 配置变更：**
- `spring.elasticsearch.rest.uris` → `spring.elasticsearch.uris`
- Repository 路径从 `com.aurora.mapper` → `com.aurora.repository`（避免与 MyBatis `@MapperScan` 冲突）

### 5.5 Swagger / Knife4j 迁移 (OpenAPI 3)

从 Swagger 2 (Springfox) 迁移到 OpenAPI 3 (Knife4j 4.x + SpringDoc)。

**注解映射：**

| Swagger 2 | OpenAPI 3 | 用法 |
|-----------|-----------|------|
| `@Api(tags = "X")` | `@Tag(name = "X")` | Controller 类 |
| `@ApiOperation("X")` | `@Operation(summary = "X")` | Controller 方法 |
| `@ApiImplicitParam(...)` | `@Parameter(...)` | 参数级注解 |
| `@ApiModel(description = "X")` | `@Schema(description = "X")` | DTO/VO 类 |
| `@ApiModelProperty(...)` | `@Schema(...)` | DTO/VO 字段 |

**AOP 切面适配：**
- `OperationLogAspect` 和 `ExceptionLogAspect` 通过反射读取注解以记录操作描述
- 代码从 `@Api.tags()[0]` / `@ApiOperation.value()` 改为 `@Tag.name()` / `@Operation.summary()`

### 5.6 RabbitMQ 异步消息

三个消费者：

| 消费者 | 功能 | 消息来源 |
|---------|------|---------|
| `MaxWellConsumer` | 监听 MySQL binlog → 同步 Elasticsearch 索引 | Maxwell 工具 |
| `CommentNoticeConsumer` | 新评论 → 发送邮件通知 | CommentService |
| `SubscribeConsumer` | 新订阅 → 发送确认邮件 | SubscribeService |

**注意：** 当前 Maxwell 驱动的 ES 同步链路（MySQL binlog → Maxwell → RabbitMQ → ES）引入了额外的中间件复杂性。一个更简洁的方案是：后端在文章变更时直接发送 MQ 消息进行 ES 同步，省去 Maxwell 依赖。该优化列入后续计划。

```
分类 ── 文章（1:N）
文章 ── 文章-标签（N:M）── 标签
文章 ── 评论（1:N）

相册 ── 照片（1:N）
评论 ── 评论（自关联，1:N：评论-回复）
资源 ── 资源（自关联，1:N：资源树）
菜单 ── 菜单（自关联，1:N：菜单树）
```

### 7.3 权限模型（RBAC）

系统采用标准的 **基于角色的访问控制（RBAC）** 模型：

```
用户 → t_user_role → 角色 → t_role_resource → 资源（API 端点）
                            → t_role_menu → 菜单（前端导航）
```

- **t_resource**：以树形结构组织，每个资源对应一个 API 端点和 HTTP 方法
- **t_role**：定义角色名称（如 admin、user、test）
- **t_user_role**：多对多关联用户与角色
- **t_role_resource**：多对多关联角色与资源权限
- **t_role_menu**：多对多关联角色与可见菜单

## 八、核心数据流

### 8.1 文章搜索同步流

```
MySQL binlog
    ↓ Maxwell 监听 binlog 变更
    ↓ 输出 JSON 格式变更事件
    ↓ RabbitMQ（maxwell Exchange → Queue）
    ↓ MaxWellConsumer.@RabbitHandler
    ↓ ElasticsearchRepository.save() / deleteById()
    ↓ Elasticsearch 索引更新
```

这是一个"双写"的搜索同步方案。文章写入 MySQL 后，通过 Maxwell（一个 MySQL binlog 监听器）将变更推送到 RabbitMQ，再由消费者同步到 ES。

### 8.2 用户登录认证流

```
1. 提交用户名+密码（或 QQ code）
2. UserAuthService 校验：
   a) 邮箱密码：BCrypt 比对
   b) QQ 登录：code → access_token → openID（调用 QQ API）
3. TokenService.createToken() 生成 JWT
4. Redis hSet 存储用户会话
5. 返回 JWT + UserInfoDTO

每次请求：
1. JwtAuthenticationTokenFilter 拦截
2. 提取 JWT → 解析 Claims
3. Redis 查用户 → 构建 Authentication
4. SecurityContextHolder 放入认证信息
5. TokenService.renewToken() 自动续期
```

### 8.3 评论通知流

```
1. 访客提交评论 → POST /comments/save
2. CommentService 保存到 MySQL
3. 发送 RabbitMQ 消息
4. CommentNoticeConsumer 异步消费
5. JavaMailSender 发送通知邮件
```

### 8.4 访客统计流

```
1. 访问页面 → POST /report
2. Redis HyperLogLog 统计 UV（去重）
3. Redis 自增 PV
4. 更新 t_unique_view 表（每日一条）
5. Quartz 定时任务每日凌晨汇总
```

### 8.5 Redis 数据结构使用

| 结构 | 用途 | Key 模式 |
|-----|------|----------|
| HyperLogLog | UV 统计（去重） | `unique_visitor:yyyy-MM-dd` |
| String | PV 计数 | `page_view` |
| ZSet | 文章排行榜 | `article_rank` |
| Hash | 用户会话 | `login_user` (field=userId) |
| BitMap | 用户签到 | `user_sign:yyyyMM` |

## 十、关键重构技术细节

### 10.1 JDK 25 兼容性问题

JDK 25 是 2025 年 9 月发布的非 LTS 版本（最新功能版本），在升级过程中遇到以下兼容性问题：

| 问题 | 原因 | 解决方案 |
|------|------|---------|
| Lombok 编译失败 | 旧版 Lombok 不支持 JDK 25 新增的 JVM 内部实现变更 | 升级到 **1.18.46**，显式配置 `annotationProcessorPaths` |
| Spring Boot 3.4.0 打包失败 | 3.4.x 内置的 ASM 版本不支持 JDK 25 class major version 69 | 升级到 **3.5.16** |
| Netty Unsafe 警告 | JDK 25 弃用 `sun.misc.Unsafe` | 不影响运行，需等待对应库的后续版本 |

### 10.2 Spring Boot 3.4.0 → 3.5.16

M1 验证中发现 Spring Boot 3.4.0 无法在 JDK 25 下运行（启动类打包后 class file version 不兼容），最终升级到 `3.5.16` 才解决问题。这也是重构方案中最初写"Spring Boot 3.4+"但在实际中必须进一步提高版本的原因。

### 10.3 Elasticsearch 命名空间迁移

```
# 旧 API（ES 7.x）
import org.elasticsearch.index.query.QueryBuilders;
import org.elasticsearch.search.fetch.subphase.highlight.HighlightBuilder;

NativeSearchQueryBuilder queryBuilder = new NativeSearchQueryBuilder()
    .withQuery(QueryBuilders.matchQuery("field", value));

# 新 API（ES 8.x）
import co.elastic.clients.elasticsearch._types.query_dsl.Query;

NativeQuery.builder()
    .withQuery(q -> q.match(m -> m.field("field").query(value)));
```

高亮 API 的迁移尤为复杂，从 `HighlightBuilder` 迁移到 `HighlightQuery + Highlight + HighlightField + HighlightFieldParameters` 四层嵌套结构。

### 10.4 OpenRewrite 自动化重构工具

在进行类似的大规模框架升级时，可以考虑使用 **OpenRewrite**（https://docs.openrewrite.org） 工具。OpenRewrite 支持自动化的源码转换 Recipe，包括：

- Spring Boot 2.x → 3.x 升级
- Jakarta 命名空间迁移
- javax.* → jakarta.* 批量替换
- 依赖版本统一升级

在 Phase 1.3（Jakarta 迁移）和 1.8（Swagger 迁移）中，使用 OpenRewrite 可以大幅减少手动替换的工作量。

## 十一、设计模式应用

### 11.1 策略模式（Strategy Pattern）

**用途 1：文件上传策略**

```java
// 策略接口
public interface UploadStrategy {
    String uploadFile(MultipartFile file, String path);
}

// 具体策略
@Component("ossUploadStrategyImpl")
public class OssUploadStrategyImpl implements UploadStrategy { ... }

@Component("minioUploadStrategyImpl")
public class MinioUploadStrategyImpl implements UploadStrategy { ... }

// 策略上下文
@Component
public class UploadStrategyContext {
    public String executeUploadStrategy(MultipartFile file, String path) {
        return uploadStrategy.uploadFile(file, path);
    }
}
```

根据配置项 `upload.mode`（oss / minio）动态选择上传策略，实现对象存储的后端无感切换。

**用途 2：文章导入策略**
支持从 `.md` 和 `.html` 文件导入文章，通过 `ArticleImportStrategy` 接口 + 不同实现类处理不同格式。

**用途 3：搜索策略（新增）**
支持 MySQL LIKE 和 Elasticsearch 两种搜索模式，通过 `search.mode` 配置切换。

### 11.2 模板方法模式（Template Method Pattern）

在策略模式的实现中，`UploadStrategy` 的多个实现类共享文件名校验、路径生成等公共逻辑，可以通过模板方法模式抽象到抽象基类中。

### 11.3 观察者模式（Observer Pattern）—— AOP 实现

通过 `@OptLog` 注解 + Spring AOP 实现操作日志的自动记录：

```java
@Aspect
@Component
public class OperationLogAspect {
    @Around("@annotation(optLog)")
    public Object around(ProceedingJoinPoint joinPoint, OptLog optLog) {
        // 前置：记录请求信息
        Object result = joinPoint.proceed();
        // 后置：异步保存操作日志到数据库
        return result;
    }
}
```

### 11.4 责任链模式（Chain of Responsibility）

Spring Security 的 `SecurityFilterChain` 本身就是责任链模式的实现：
```
JwtAuthenticationTokenFilter → UsernamePasswordAuthenticationFilter → ... → Controller
```

### 11.5 工厂模式（Factory Pattern）

通过 Spring DI 自动注入策略实现类到 `StrategyContext`，由 Context 根据运行配置选择具体策略。

### 11.6 设计模式使用总结

| 模式 | 用途 | 实现位置 |
|------|------|----------|
| 策略模式 | 文件上传、文章导入、搜索 | `strategy/` 包 |
| 模板方法 | 上传策略的公共逻辑抽象 | `strategy/UploadStrategy` 体系 |
| 观察者/AOP | 操作日志、异常日志自动记录 | `@OptLog` + `aspect/` |
| 责任链 | 请求认证与鉴权 | `SecurityFilterChain` + `JwtAuthenticationTokenFilter` |
| 工厂模式 | 策略实例的创建与选择 | 通过 Spring DI 注入 |

## 十四、遗留问题与改进方向

### 14.1 已识别但未解决的问题

| 问题 | 优先级 | 说明 |
|------|--------|------|
| `authorizeRequests` 废弃警告 | 低 | `WebSecurityConfig` 仍需从 `authorizeRequests` 迁移到 `authorizeHttpRequests` |
| `selectCount` Long 转 Integer | 低 | 数据量超过 `Integer.MAX_VALUE` 时需改为 Long |
| Lombok/Netty Unsafe 警告 | 低 | JDK 25 弃用 `sun.misc.Unsafe`，不影响运行 |
| UserAgentUtils 未完全替换 | 中 | pom.xml 同时保留 `UserAgentUtils 1.20` 和 `yauaa 7.28.1`，实际代码仍需迁移 |
| ip2region 未升级 | 低 | 保持 1.7.2，API 兼容 |
| 配置占位符 | 中 | `application-prod.yml` 中的 `${mysql的ip}` 等需部署前替换 |
| ES xpack.security | 中 | ES 8.x 默认启用安全认证，本地开发需关闭 |
| 后台 Vue 2 技术栈 | 高 | 需迁移到 Vue 3 + Element Plus + Pinia |

### 14.2 架构层面的优化建议

**ES 同步链路简化：**
当前 `MySQL → Maxwell → RabbitMQ → ES` 的同步链路引入了 Maxwell 额外组件。建议改为后端 Service 层在文章变更时直接发送 MQ 消息进行 ES 同步，将链路缩短为 `Service → RabbitMQ → ES`。

**后台 Vue 3 迁移：**
将后台从 Vue 2 + Element UI + Vuex 迁移到 Vue 3 + Element Plus + Pinia，与前台技术栈统一，降低维护成本。

**Virtual Threads 应用：**
JDK 25 对 Virtual Threads 的支持已非常成熟，可在 Service 层 I/O 密集型操作（数据库查询、ES 搜索、文件上传）中启用 Virtual Threads 提升吞吐量。

**OpenRewrite 自动化：**
未来类似的大规模框架升级可引入 OpenRewrite 工具，通过声明式 Recipe 自动执行源码转换。
