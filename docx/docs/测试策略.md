# Aurora 重构版测试策略

> 本文档定义重构项目的测试范围、策略和计划。重构项目风险较高，测试是防回归的核心保障。

---

## 一、测试分层策略

```
        /\             
       /  \            
      / E2E \          ← 端到端测试（少量，覆盖核心流程）
     /--------\        
    / 集成测试  \       ← 中等数量，覆盖 API 层 + 数据库交互
   /--------------\    
  /  单元测试       \   ← 大量，覆盖 Service 层核心逻辑
 /__________________\  
```

| 层级 | 覆盖目标 | 数量级 | 运行时间 | 框架 |
|------|---------|--------|---------|------|
| 单元测试 | Service 层核心方法、工具类 | ~80% 覆盖率 | 秒级 | JUnit 5 + Mockito |
| 集成测试 | Controller API、数据库交互 | 每模块 2-5 个 | 分钟级 | @SpringBootTest + MockMvc |
| E2E 测试 | 登录→发文章→搜索全流程 | 1-3 个 | 分钟级 | TestRestTemplate / Playwright |

## 二、单元测试计划

### 2.1 优先级 P0（必须覆盖）

| 类 | 测试内容 | 说明 |
|----|---------|------|
| TokenServiceImpl | createToken, parseToken, refreshToken, renewToken, delLoginUser | JWT 操作核心，重构中 API 变化最大 |
| UserAuthServiceImpl | register, login (邮箱+QQ), updatePassword, logout | 登录认证核心 |
| ArticleServiceImpl | listArticles, getArticleById, saveOrUpdateArticle, listArticlesBySearch | 文章业务核心，涉及 ES |
| RedisServiceImpl | hSet, hGet, hDel, sSet, sGet, 封装方法 | Redis 操作封装 |

### 2.2 优先级 P1（重要）

| 类 | 测试内容 | 说明 |
|----|---------|------|
| CommentServiceImpl | saveComment, listComments, listRepliesByCommentId | 评论 + MQ 通知 |
| CategoryServiceImpl | listCategories, saveOrUpdateCategory | 分类 CRUD |
| TagServiceImpl | listTags, saveOrUpdateTag | 标签 CRUD |
| AuroraInfoServiceImpl | getAuroraHomeInfo, getAuroraAdminInfo | 首页/后台信息聚合 |

### 2.3 测试示例（TokenServiceImpl）

```java
@ExtendWith(MockitoExtension.class)
class TokenServiceImplTest {

    @InjectMocks
    private TokenServiceImpl tokenService;

    @Mock
    private RedisService redisService;

    @Test
    void shouldCreateTokenWhenGivenUserId() {
        String token = tokenService.createToken("1");
        assertNotNull(token);
        assertTrue(token.startsWith("eyJ")); // JWT header
    }

    @Test
    void shouldParseTokenWhenGivenValidToken() {
        String token = tokenService.createToken("1");
        Claims claims = tokenService.parseToken(token);
        assertEquals("1", claims.getSubject());
    }
}
```

## 三、集成测试计划

### 3.1 测试策略

- 使用 `@SpringBootTest(webEnvironment = SpringBootTest.WebEnvironment.RANDOM_PORT)`
- 使用 H2 内存数据库或 Testcontainers（MySQL）作为数据库
- Redis/MQ/ES 使用 Mock 或 Testcontainers
- 每个 Controller 至少一个成功场景 + 一个异常场景

### 3.2 覆盖范围

| Controller | 测试场景 |
|-----------|---------|
| ArticleController | 获取置顶文章、搜索文章、发布文章、删除文章、密码访问 |
| UserAuthController | 注册、登录、发送验证码、QQ 登录 |
| CommentController | 提交评论（含限流）、获取评论列表 |
| CategoryController | CRUD + 分页查询 |
| AuroraInfoController | 获取首页/后台信息、更新配置 |

### 3.3 测试示例（ArticleController）

```java
@SpringBootTest
@AutoConfigureMockMvc
class ArticleControllerTest {

    @Autowired
    private MockMvc mockMvc;

    @Test
    void shouldReturnArticlesWhenListArticles() throws Exception {
        mockMvc.perform(get("/articles/all"))
                .andExpect(status().isOk())
                .andExpect(jsonPath("$.code").value(200));
    }

    @Test
    void shouldReturnArticleByIdWhenExists() throws Exception {
        mockMvc.perform(get("/articles/1"))
                .andExpect(status().isOk())
                .andExpect(jsonPath("$.data.articleTitle").isString());
    }
}
```

## 四、Mock 策略

| 中间件 | Mock 方式 | 说明 |
|--------|---------|------|
| Redis | Mock RedisService 接口 | 不启动真实 Redis |
| Elasticsearch | Mock ElasticsearchMapper | 不启动真实 ES |
| RabbitMQ | Mock RabbitTemplate + @SpringBootTest 排除 | 不启动真实 MQ |
| MySQL | H2 内存数据库 或 Testcontainers | 单元测试用 H2，集成测试用 Testcontainers |
| OSS/MinIO | Mock UploadStrategyContext | 不上传真实文件 |

## 五、测试与重构的配合节奏

```
Phase 1（框架升级）→ 先跑通现有测试（如有），再改代码
                     → 每完成一个模块的升级，手动验证关键接口
Phase 2（代码治理）→ 补充单元测试和集成测试
                     → 先写测试再重构（TDD 风格）
Phase 3（前端重构）→ 前端使用 Vitest 测试工具函数
                     → 组件测试选核心组件开始
Phase 4（基础设施）→ 在 CI/CD 中集成 mvn test
```

## 六、测试质量检查清单

- [ ] 所有 P0 类有单元测试
- [ ] 所有 Controller 有集成测试
- [ ] 测试覆盖率达到 60%+（Service 层）

## 七、自动化验证脚本

> 每次 Part 完成后，通过 [verify.ps1](./verify.ps1) 脚本一键完成编译、测试、启动、接口冒烟。

### 7.1 前置要求

运行全量验证需要以下基础设施（推荐 Docker 运行）：

| 服务 | 必需？ | 无此服务的后果 |
|------|--------|---------------|
| MySQL | 必需 | 后端启动失败 |
| Redis | 必需 | 后端连接超时失败 |
| RabbitMQ | 必需 | 后端启动阶段连接失败 |
| Elasticsearch | 非必需 | 搜索接口不可用，其余功能正常 |

### 7.2 脚本使用方式

```powershell
cd 重构版/

# 全量验证（M1 回归时用）
.\verify.ps1

# 验证特定 Part（如 Part 1.4）
.\verify.ps1 1.4
```

脚本执行流程：
1. 检查基础设施（自动检测 Docker 是否可用）
2. mvn clean compile 编译检查
3. mvn test 或 mvn package + 单元测试
4. 启动后端（后台进程 java -jar）
5. 依次调 5 个核心 GET 接口
6. 尝试搜索接口（ES 可用时才测）
7. 停止后端

### 7.3 与 Docker 的关系

| 验证步骤 | 需要 Docker | 需要中间件 |
|---------|------------|-----------|
| mvn clean compile | 否 | 无 |
| mvn test | 否 | 无（Mock 化） |
| mvn package | 否 | 无 |
| 启动 java -jar | 是 | MySQL + Redis + RabbitMQ |
| 接口冒烟 | 是 | MySQL + Redis |

### 7.4 与本次重构的配合

- Phase 1 前期（Part 1.1-1.3）：只做编译验证，跳过启动
- Phase 1 后期（Part 1.4-1.8）：每 Part 跑 .\verify.ps1
- M1 里程碑：全量验证全部通过
- Phase 2-4：每 Part 至少 mvn test
