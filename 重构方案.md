## 三、JDK 25 兼容性分析与依赖调整

### 3.1 必须升级的依赖

| 依赖 | 当前版本 | 方案 | 新版本 | 原因 |
|------|----------|------|--------|------|
| Elasticsearch Server | 7.9.2 | 升级 | 8.15.x | bundled JDK 14/15 不支持 JDK 25 |
| 阿里云 OSS SDK | 3.8.0 | 升级 | 3.17.x+ | 版本太老 |
| Lombok | 内置 | 显式指定 | 1.18.34+ | JDK 25 需要新版 Lombok |
| fastjson | 1.2.76 | 替换 | Jackson 或 fastjson2 2.0.53+ | 安全漏洞 |
| Hutool | 5.7.5 | 升级 | 5.8.32+ | JDK 25 兼容 |
| UserAgentUtils | 1.20 | 替换 | yauaa | 停更于 2015 年 |
| MyBatis-Plus | 3.4.2 | 升级 | 3.5.9+ | JDK 25 适配 |
| Knife4j | 2.0.7 | 升级 | 4.5.0+ 或 SpringDoc | 不兼容 Spring Boot 3.x |
| Commons Codec | 1.8 | 升级 | 1.17+ | 版本偏老 |
| ip2region | 1.7.2 | 升级 | 2.x | 建议升级 |
| jjwt | 0.9.0 | 升级 | 0.12.x | API 过时 |
| MySQL Connector | 8.0.x | 升级 | 9.1.0+ | 适配 MySQL 9.x |

### 3.2 预计兼容（无需改动）

Spring Boot 3.4+、Spring Security 6.x+、Redis Lettuce、RabbitMQ Client、Quartz、Maven 3.9.x

### 3.3 Elasticsearch 升级要点

- ES 7.9.2 与 JDK 25 冲突是最大障碍，必须升级到 ES 8.15.x
- IK 分词器同步升级（版本必须一致）
- Spring Data Elasticsearch 随 Spring Boot 3.4 自动升级
- ES 8.x 默认启用 xpack.security，本地开发需关闭
- 本机 9200 端口被 Cpolar 占用，ES 需改端口 9201 或关闭 Cpolar

### 3.4 本机环境适配策略（不降级、可升级、JDK 锁定 25）

## 四、重构原则

1. **保持业务功能不变**：重构期间不新增业务功能，只升级技术栈与优化代码结构
2. **原始项目只读**：所有修改在 重构版/ 目录进行，aurora/ 作为对比基线
3. **分阶段提交**：每个 Part 完成后单独提交 Git，便于回滚
4. **先让项目跑起来，再优化**：先完成核心升级，再进入代码治理
5. **避免过度设计**：不引入不必要的抽象

## 七、风险与依赖

| 风险 | 影响 | 应对措施 |
|------|------|----------|
| Spring Boot 3.x 与 Spring Security 6.x 改动大 | 高 | 参考官方迁移指南，逐模块验证 |
| JDK 25 为新版本，部分第三方库尚未完全适配 | 中 | 优先选择已支持 JDK 25 的依赖版本 |
| Elasticsearch 7.9.2 不支持 JDK 25，必须升级 8.x | 高 | 同步升级 ES 服务端、IK 分词器、Spring Data ES Client |
| javax jakarta 涉及文件多 | 高 | 使用 IDE 全局重构，结合编译报错逐个修复 |
| fastjson 替换可能影响序列化行为 | 中 | 重点测试时间格式、空值处理、泛型反序列化 |
| UserAgentUtils 等停更库需替换 | 中 | 提前验证替代库 API，确保行为一致 |
| 前端 Vue 2 Vue 3 迁移工作量大 | 高 | 作为独立 Phase，可放到后端稳定后再进行 |
| 测试覆盖不足，重构后易引入回归 | 高 | 先补充核心流程测试，再进行大改动 |
